Project to project linkage

In a typical Linkage Project, records will only link to other records within that Linkage Project, with Linkage Projects completely independent from each other.

LinXmart also supports linkage between Linkage Project, referred to as Project-to-Project linkages.

A common use of this type of linkage is when:

  • one Linkage Project contains an ongoing repository of data that is regularly updated, and
  • there is a need to link a new dataset to some or all of this data, but
  • this new dataset is not to become part of this ongoing repository.

An example is shown in below. Here the Linkage Project NSW Master Map is a repository of health-based information from NSW that is regularly updated. A researcher who holds a separate registry cohort dataset (the NSW Bowel Surgery Registry) wishes to link their data to mortality information to investigate mortality rates in this cohort. However, this registry is not to become part of the NSW Master Map; it is solely being used in one research project. By utilising project-to-project linkage, this registry data can be linked against the NSW Master Map without joining or modifying it.

Example of project-to-project linkage; a registry cohort is linked against data in an ongoing repository without joining the repository

info

In this section we refer to the ongoing repository as the "reference project", and the registry the "calling project".
Note that LinXmart does not support the chaining of Linkage Projects. That is, if Project A is linked to Project B, then Project B cannot be linked to Project C. Chaining or cascading Linkage Projects in this way will produce unexpected results.

Configuring project-to-project linkage​

The configuration of a project-to-project linkage is established when the calling project is created and a project-to-project linkage is declared. The operator declares the reference project that is to be linked to, and then declares which event types from the reference project are to be included in the linkage.

To configure project-to-project linkage, select the calling project from the PROJECTS tab to bring up the Project Details screen. From here, select the Project Links button from the menu in the top right-hand corner. Linking to one or more Event Types in another Linkage Project is achieved by clicking Add Project Link in the Project-to-Project Linkages panel.

The Add Project Links page will be displayed. From the Link To drop-down box, select the reference Linkage Project. The Event Types that are attached to this project will be listed. Select all those that you wish this project to link and click Save to complete.

Matching and grouping​

During the Matching phase of a project-to-project linkage, LinXmart looks at the project link configuration of the calling project and identifies the Event Type(s) in the reference project that are to be linked. All the reference project records that are of the specified Event Type(s) are included in the matching. For example, if calling project A links to Event Types X and Y in reference project B, then Project B’s event type X and Y records are included in the Matching phase of Project A.

The Matching phase will produce a number of match pairs that link records in the calling project (A) to records in the reference project (B). The Grouping phase for Project A uses these pairs, along with the groups in Project B, to group events for Project A. This means that two events in a calling project can be grouped through indirect association to a group in the reference project.

Records can be brought together indirectly through Project-to-Project linkage

info

In project-to-project linkage, it is important to note that no records, pairs or groups in the reference project are in any way altered or affected by the linkage with the calling project. Any links that are made by the calling project to records in the reference project, and any resulting groups or pairs are all stored in the calling project.

Merge grouping​

The merge grouping process will process every matching pair of records with a weight above the defined threshold and form groups of all the records that are linked via the pairs.

In project-to-project linkage, records in different groups in a reference project may come together into a single group in the calling project. For example, the figure below shows the results of a grouping where records 2, 3 and 4 in calling project A link to each other and to records in different groups in reference project B. Merge grouping will bring all of the records (2, 3, 4) from project A and (10, 13 and 14) from Project B into a single group in Project A, even though record 10 is in a different group to records 13 and 14 in Project B.

Merge grouping in Project-to-Project linkage can bring together records from the reference project that exist there in separate groups

Weighted best-link grouping​

Weighted best-link first orders all match pairs by their weight in descending order, so that those with the highest score are processed first. When a calling project uses this method to link to a reference project, it honours the groups in the reference project, meaning these groups cannot be merged together.

The figure below shows the same records and pairs as in the above Merge grouping example. Note that the resulting groups are different. As each matching pair is processed, the weighted best-link grouping strategy decides that if that pair would break any rules it is ignored, as illustrated by the dotted lines. These rules prevent existing groups in either Project A or Project B from coming together and prevent redundant pairs from being processed. In this example, two groups are formed: records 2 and 10 in one group, and records 3, 4, 13 and 14 in another.

The same pairs result in different groupings when using Weighted Best-Link Grouping for Project-to-Project linkage

Updates to the reference project​

The previous section discussed the matching and grouping which occur when records are added to the calling project. However, LinXmart also makes changes to the calling project when records are added, amended or deleted from the reference project.

When records are added, deleted or amended in a reference project, LinXmart will initiate secondary jobs to update the linkage maps of any calling project(s) that are affected. This keeps reference projects and any dependencies (calling projects) synchronised. The secondary jobs spawned by LinXmart are visible in the Jobs screen of the Web Interface and appear as extra jobs in the incoming Envelope Request.
